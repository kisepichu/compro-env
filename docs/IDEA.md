まず、設計手法については、ドメイン駆動設計を人間がやるような手順で進めたい。人間は要件定義、ドメインモデル図やユースケース図などを書いていくが、そのようなものを AI を用いた開発ではどうやるか知らないので、 https://github.com/shanraisshan/claude-code-best-practice やネット記事等を調べて予定を立てる。最初に issue を claude code が触れるようにする、毎日自動起動、許可の必要がある場所、日報の仕組み等、自走のための環境構築をしたいからそこら辺を教えてもらう。

競プロ環境をきれいに作り直したい。まず一般の人向けの似たツールの実装として https://github.com/online-judge-tools/oj がある。これに自分用の機能(コンテスト用ディレクトリ管理、コード整形、入力コード自動生成など)を追加したものを以前作り使っていた。 ref-repos に参考リポジトリ 3 つへのシンボリックリンクを置いた(library 以外の二つ。)(内容見れるか? 見れないなら報告。)。ac が実際に使っていた python 製でコンテスト内 C++ 用のもので、 giming は Rust で書き直そうして骨格だけ考えていたものである。これらを参考にし、新しく Rust でコンテスト内の言語も Rust 用に書き直したい。というかコンテスト内の言語は設定で変えられるようにして、テスト方法とか提出前のコード整形機能とかもコマンドをカスタマイズできるようにしたい。例えばテストは C++ はコンパイルして標準入力にそのまま入れる感じだったが、 Rust では言語機能として test があるからそれを使うようにユーザーが config に cargo test コマンド等を書いて設定できるようにする。問題ごとにも言語変えられるようにしたいな。あと入力自動生成は Tera のコンテキストに問題から読み取ったものうまく突っ込んで、ユーザーが書いた template レンダリングするだけ(例えば test はコード下部の solve とか使って書くテンプレートに sample input や output 等を流し込む)でできるようにするとかできるといいな。入力は複雑な配列とかあるし自由な形式だからそれを Tera で表現してうまくできるかわからんけど。検討必要。
基本は ac 内にあった機能が全てあればいいが、ソースファイルを一個だけ使いまわしていたことをやめて問題ごとにフォルダを作ってその中にさらに解法フォルダをいくつか作り、コンテスト後の復習や別言語での回答等もできるようにするなど、改良できることはしたい。あと CLI ツールにして、 sub とかのこのツールのコマンド以外が入力されたら普通にシェルコマンドとして実行するようにするとか、 giming で予定していた機能も入れたい。ここら辺は仕様をまとめる時にちゃんと考える
giming リポジトリを書いたときは、ドメイン駆動設計やクリーンアーキテクチャを学び始めで、頑張ってディレクトリ構造とかドメインを考えていたもので、設計の考え方的に間違っていたりベストプラクティスでない可能性がある。今回も設計をしっかりしたいし、僕が設計を学びながら作りたいので、 giming リポジトリのものをイメージとして、さらに正確にドメインを考えてアーキテクチャもしっかりしたい。設計をする段階で、ドメインモデルができた後、 giming のディレクトリ構造やコードの詳細をまず見て、「実践ドメイン駆動設計(Vaughn Vernon の Event Storming)」などを参考にしてそれがドメイン駆動設計の観点からベストプラクティスに沿っているか確認し、沿っている場合はそのままコピーでよく、そうでない場合は人間にどこがどう沿っていないか説明して、新しく構造を組み直す。命名やディレクトリ構造は人間の介入が必要なので、一旦説明、議論必要。
ac を使っていた当時から変わったこととして、まず最近 AtCoder が bot 対策でログインに Captcha 的なものを導入したから、その方法は使えないのをどう解決するか検討する必要がある。(oj はどうやっているんだろう? 調べて予定立てたい。ブラウザから Cookie 取ってきてもらい張るとかを giming で試していた気がするが忘れた。) また、 nvim を使い始めたので、エディタとの連携は nvim で必ず動くようにし、 vscode などもオプションで使えるようにする。あ、 vscode を使っていたときは、例えば sub a で A 問題に submit コマンドをしたら、自動で code (b のファイル) が実行されて、 vscode の b 問題のコードのタブが自動で選択されてたけど、 nvim だとターミナルで nvim (ファイル) とするとそこで開いちゃってできないな。今開いてるウィンドウの中でバッファを開く的な方法を知る必要もある。
あと、 ac でなかった機能として、ライブラリ機能も統合したい。前使っていた library は参考リポジトリ ref-repos/library にある。またいい感じの他の人のライブラリとして https://github.com/to-omer/competitive-library/tree/master?tab=readme-ov-file も参考。
この競プロ環境プロジェクト内のコードとして、 solutions フォルダは giming/solutions と同じような感じで作り、別で library フォルダも作り、例えば lib/rust フォルダに rust 用ライブラリをまとめる感じ。 library/rust/Cargo.toml があり library/rust/crates/library と library/rust/crates/tools (ジャッジ用や他の仕組みに使う?) みたいな感じイメージ。 crates/library 内のライブラリの性質ごと(algebra とか)はフォルダ分けだけで対応。他の言語のライブラリは library/cpp/ に作り、各言語でふさわしい感じで library と tools などを作る。そして、 verify についてだが、よくあるライブラリだとライブラリコードに付随して AOJ などの問題の解法で verify 用コードを書き、実際にオンラインジャッジに提出して verify する方法。今回は提出したり過去のコードも管理するアプリと統合するので、過去のコードを再利用/同じように解法コードを書きライブラリ側に独自記法などで「この解法を verify として使う」と記載する仕組みを各言語について作る。(各言語の tools/verify と、この競プロ環境プロジェクトの方のコードで実現?) 競技中とか解法コードの方では、 include が普通にできるように工夫する。そして、 submit の前のコード整形の際に、一つの提出ファイルにまとまるようにライブラリを埋め込む仕組みも各言語について作る必要がある(tools/format ? tools とかいう名前とかもわかりやすくできないか検討。上のコードから使われるから普通に lib とかか? そうするとメインの library とかぶるんだよな。)。 rust はそのような crate があった気がするからうまく使ってもいい。
そして、 to-omer のライブラリのような、ライブラリコードを公開するための web ページも作りたい。 to-omer は cargo の doc +改変? で実現しているが、このプロジェクトでは複数言語のライブラリをまとめるので、独自で verify ごとに生成するようにして github pages に自動的に上がる仕組みを考える。 ref-repos/library でやっていたような、 markdown でライブラリに説明をつけられる仕組みも入れたい。ライブラリコード内に doc のパスを指定?
その web ページでは、ライブラリを言語ごと/分野ごとにフォルダ間を移動でき、各ファイルに説明とコード、 verify 結果(バッジ?)、できたら verify コードへのリンク(通常のコンテストの解法コードもこれで公開してもいい。生成して静的においておくだけだとサイズがでかくなりそう?、 github pages だけで、見たときにうまく github からコードなどを引っ張って来るようにする)を置く。解法コードに説明や考えたことつけられるのややアツいから一旦後回しにして optional で md に書いたり issue とかにしていつか実装か。
イメージしていることを大体書いたつもりだがまだ足りない気がするので質問してもらう
